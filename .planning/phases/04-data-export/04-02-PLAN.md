   ```typescript
   if (!response.ok) {

   }
   ```

5. Convert streaming response to blob:
   ```typescript

   ```

6. Extract filename from Content-Disposition header:
   ```typescript
   const contentDisposition = response.headers.get('Content-Disposition')
   let filename = 'logs.csv'  // fallback
   if (contentDisposition) {
     const matches = /filename=([^;]+)/.exec(contentDisposition)
     if (matches?.[1]) {
       filename = matches[1].trim()
     }
   }
   ```

7. Trigger browser download:
   ```typescript
   const downloadUrl = URL.createObjectURL(blob)
   const link = document.createElement('a')
   link.href = downloadUrl
   link.download = filename
   document.body.appendChild(link)
   link.click()

   // Cleanup
   document.body.removeChild(link)
   URL.revokeObjectURL(downloadUrl)
   ```

Why this approach:
- Reuses filter parameter building from fetchLogs (consistency, DRY principle)
- Blob + createObjectURL pattern is standard for file downloads (per RESEARCH.md Pattern 3)
- Content-Disposition parsing respects backend filename pattern (logs-YYYY-MM-DD-HHMMSS.csv)
- Error handling matches existing API client pattern (throw Error with detail message)
- Cleanup prevents memory leaks (revokeObjectURL releases blob memory)

Why search parameter excluded:
- Backend /api/export endpoint doesn't implement full-text search (only list endpoint does)
- Export focuses on structured filters (severity, source, date range) per CONTEXT.md decisions
- If search is active, user sees subset in UI and can export that visible subset using other filters

Create new Client Component for export button with loading and error states:

1. Component file: `frontend/src/app/logs/_components/export-button.tsx`

2. Component structure:
   ```typescript
   'use client'

   import { useState } from 'react'
   import { Download } from 'lucide-react'
   import { Button } from '@/components/ui/button'
   import { toast } from 'sonner'
   import { exportLogs } from '@/lib/api'
   import { LogFilters } from '@/lib/types'

   interface ExportButtonProps {
     filters: LogFilters
     disabled?: boolean  // Disable when no logs match filters
   }

   export function ExportButton({ filters, disabled }: ExportButtonProps) {
     const [isExporting, setIsExporting] = useState(false)

     const handleExport = async () => {
       setIsExporting(true)

       try {
         await exportLogs(filters)
         toast.success('CSV exported successfully', {
           description: 'Your file has been downloaded'
         })
       } catch (error) {
         toast.error('Export failed', {
           description: error instanceof Error ? error.message : 'Unknown error'
         })
       } finally {
         setIsExporting(false)
       }
     }

     return (
       <Button
         onClick={handleExport}
         disabled={disabled || isExporting}
         variant="outline"
       >
         <Download className="mr-2 h-4 w-4" />
         {isExporting ? 'Exporting...' : 'Export CSV'}
       </Button>
     )
   }
   ```

3. Component behavior:
   - Props: `filters` (current LogFilters state), `disabled` (optional, for empty results)
   - State: `isExporting` boolean for loading state
   - Click handler: async function that calls exportLogs with filters
   - Success: Show toast "CSV exported successfully" with description "Your file has been downloaded"
   - Error: Show toast "Export failed" with error message from API
   - Loading: Button disabled with text "Exporting..." (spinner implicit in button text change)
   - Button always shows Download icon (lucide-react)

Why these choices:
- Client Component ('use client') required for useState and event handlers
- Toast notifications match Phase 3 pattern (Sonner already integrated)
- variant="outline" provides secondary button styling (not primary action)
- Download icon per CONTEXT.md decision ("Button displays 'Export CSV' with download icon")
- disabled prop allows parent to disable when no logs (per CONTEXT.md: "Button disabled when no logs match filters")
- Error handling provides specific message from API (helps user debug filter issues)
- isExporting state prevents double-clicks during export

Why "Exporting..." text:
- CONTEXT.md specifies "Loading state: button disabled with spinner during export"
- Text change provides feedback while disabled state prevents interaction
- Simpler than adding separate spinner component (button text conveys state clearly)
