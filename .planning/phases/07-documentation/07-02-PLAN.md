**Required sections:**

1. **Header:**
```markdown
# ADR-004: Use timestamptz with UTC Normalization for Timezone Correctness

**Status:** Accepted
**Date:** 2026-03-20
**Deciders:** Development Team
```

2. **Context:**
- Problem: Log timestamps must preserve timezone information and aggregate correctly across time zones
- Technical context: PostgreSQL 18, Python datetime with timezone awareness, JavaScript Date objects in frontend
- Real-world scenario: Logs generated in different timezones (server in UTC, developer in PST, user in JST)
- Specific issues with naive timestamps:
  - "3 PM" could mean 3 PM UTC, 3 PM PST, or 3 PM JST - ambiguous
  - Analytics aggregations produce incorrect results when mixing timezones
  - Date range filters break (filter "March 20" but which timezone's March 20?)
- Requirement: ANALYTICS-07 mandates explicit timezone handling for time-series aggregations

3. **Requirements Addressed:**
```markdown
- **DB-02:** Timestamp column uses timestamptz (timezone-aware) data type
- **ANALYTICS-07:** Time-series aggregations use explicit timezone handling
- **NFR-Correctness:** No timezone bugs in date range queries or analytics
- **API-07:** All API endpoints include input validation (reject timezone-naive timestamps at schema level)
```

4. **Options Considered:**

**Option 1: Naive Timestamps (timestamp without time zone)**
- Pros: Simpler storage, smaller data type (8 bytes), no timezone conversion needed
- Cons: Ambiguous meaning ("3 PM" in which timezone?), analytics produce wrong results, date range queries break, cannot represent DST transitions correctly

**Option 2: Timestamp with Local Timezone Storage (timestamptz storing original)**
- Pros: Preserves original timezone information, can display "as entered"
- Cons: Aggregations complex (need to normalize to single timezone), query filters require timezone conversion, cannot sort chronologically across timezones

**Option 3: Timestamp with UTC Normalization (timestamptz storing UTC)**
- Pros: Unambiguous meaning, correct aggregations (all in UTC), chronological sorting works, date_trunc() produces correct buckets, standard industry practice
- Cons: Original timezone information lost (only UTC stored), display requires client-side conversion, slightly more complex input validation

**Option 4: Store Both (timestamptz + timezone name column)**
- Pros: Preserves original timezone, enables UTC aggregations
- Cons: Storage overhead (16 bytes + varchar), complex synchronization (two columns must stay consistent), overkill for log viewing use case

5. **Decision:**
```markdown
We will use timestamptz column type with UTC normalization:

1. **Database column:**
   - Column type: TIMESTAMP(timezone=True) in SQLAlchemy → timestamptz in PostgreSQL
   - Store all timestamps in UTC (converted on insert if needed)
   - PostgreSQL automatically converts to UTC on storage

2. **API validation:**
   - Pydantic schema requires timezone-aware datetime (validator rejects timezone-naive)
   - Reject requests with timestamps like "2024-03-20T15:30:00" (no timezone)
   - Accept ISO 8601 with timezone: "2024-03-20T15:30:00+00:00" or "2024-03-20T15:30:00-08:00"

3. **Analytics queries:**
   - Use date_trunc() with explicit timezone: date_trunc('hour', timestamp AT TIME ZONE 'UTC')
   - All aggregations performed in UTC (consistent bucketing)
   - Frontend converts to local timezone for display (JavaScript Date.toLocaleString())

4. **Frontend handling:**
   - datetime-local input with setValueAs converting to ISO 8601 with timezone
   - Display timestamps in browser's local timezone (automatic via JavaScript)
   - API calls always send timezone-aware ISO 8601 strings
```

6. **Consequences:**

**Positive:**
- Unambiguous timestamp meaning (always UTC in database)
- Correct analytics aggregations across all timezones
- Date range filters work correctly (compare UTC to UTC)
- Follows PostgreSQL best practices (timestamptz is recommended type)
- Pydantic validation catches timezone-naive timestamps at API boundary
- Standard industry pattern (AWS CloudWatch, Datadog, Splunk all use UTC storage)

**Negative:**
- Original timezone information lost (cannot show "entered at 3 PM PST")
- Requires frontend to handle timezone conversion for display
- Slightly more complex API validation (Pydantic field validator needed)
- Developers must understand UTC vs local time concepts

**Neutral:**
- PostgreSQL timestamptz always stores UTC internally (automatic conversion)
- JavaScript Date objects handle timezone display automatically
- ISO 8601 with timezone is standard format (no custom parsing needed)

7. **Alternatives Rejected:**

**Naive Timestamps:** Rejected because analytics aggregations produce incorrect results. Example: Log at "2024-03-20 15:00:00" (no timezone) + Log at "2024-03-20 15:00:00" (no timezone) appear identical but might be 8 hours apart if one is UTC and one is PST. Requirement ANALYTICS-07 mandates explicit timezone handling - naive timestamps violate this.

**Local Timezone Storage:** Rejected because aggregating across timezones is complex. Example: date_trunc('hour', timestamp) produces different buckets depending on original timezone - logs at "3 PM PST" and "3 PM JST" are 17 hours apart but bucket to same hour. Correct aggregation requires converting all to single timezone (UTC) first - simpler to store as UTC.

**Store Both Columns:** Rejected because storage overhead (16 bytes + varchar vs 8 bytes) isn't justified for log viewing use case. Timezone name needed for "entered at 3 PM PST" display, but users don't need this level of detail. Adds complexity (two columns must stay synchronized) with minimal UX benefit. Deferred to v2 if user research shows demand.

- Schema validation: `backend/app/schemas/logs.py` (LogCreate with timezone validator)
- Analytics queries: `backend/app/routers/analytics.py` (date_trunc with AT TIME ZONE 'UTC')

Create docs/decisions/005-frontend-architecture.md following the extended ADR template. This ADR covers multiple related frontend patterns and principles.

**Required sections:**

1. **Header:**
```markdown
# ADR-005: Frontend Architecture Patterns and Design Principles

**Status:** Accepted
**Date:** 2026-03-21
**Deciders:** Development Team
```

2. **Context:**
- Problem: Need consistent frontend patterns for Next.js 15 with App Router, balancing SSR performance with client-side interactivity
- Technical context: Next.js 15.5.14, React 19.2.4, TypeScript 5.9.3, App Router architecture
- Multiple architectural decisions needed:
  - When to use Server Components vs Client Components
  - How to extract reusable logic (custom hooks vs utility functions)
  - Where to store UI state (URL vs React state vs context)
  - How to structure component composition (container vs presentation)
- Requirement: Demonstrate technical excellence with clean patterns suitable for portfolio evaluation

3. **Requirements Addressed:**
```markdown
- **UI-05:** Frontend uses React Server Components for data fetching
- **UI-06:** Frontend uses Client Components only for interactive features
- **FILTER-07:** Filter state persists when navigating between pages (URL state)
- **NFR-Maintainability:** Consistent patterns enable easy feature additions
- **NFR-Performance:** SSR with selective hydration for fast initial page load
```

4. **Topics Covered:**

**Topic 1: Server Components vs Client Components Strategy**

**Context:** Next.js 15 App Router defaults to Server Components. Decision needed: when to use 'use client' directive?

**Options Considered:**
- Option A: Client Components everywhere (traditional React SPA approach)
- Option B: Server Components everywhere with Server Actions for interactivity
- Option C: Hybrid - Server Components for data fetching, Client Components for interactivity only

**Decision:** Hybrid approach following Next.js 15 best practices.

**Rules:**
1. Server Components for data fetching and initial render (pages, layouts)
2. Client Components for interactivity (forms, modals, infinite scroll)
3. Prop drilling from Server to Client Components for initial data
4. Client Components fetch additional data via API routes (not Server Actions for UI state)

**Examples:**
- Server Component: `app/logs/page.tsx` (fetches initial logs, renders initial HTML)
- Client Component: `_components/log-list.tsx` (manages infinite scroll, filter state)
- Server Component: `app/analytics/page.tsx` (fetches initial analytics, SSR charts)
- Client Component: `_components/time-range-filter.tsx` (interactive date picker)

**Benefits:** Fast initial page load (SSR), minimal JavaScript shipped to browser, SEO-friendly, progressive enhancement

---

**Topic 2: Custom Hooks Pattern (When to Extract Logic)**

**Context:** Reusable logic appears across multiple components. Decision needed: when to extract to custom hook vs keep inline?

**Options Considered:**
- Option A: Extract all logic to hooks (max reusability)
- Option B: Keep logic inline in components (simplicity)
- Option C: Extract when logic is reused 2+ times OR complex enough to test independently

**Decision:** Extract to custom hook when meets criteria: (1) Used in 2+ components, OR (2) Complex enough to warrant independent testing, OR (3) Represents a distinct concern (state management, data fetching, side effects).

**Examples of custom hooks created:**
- `use-infinite-scroll.ts` - Manages cursor pagination state and data fetching (complex state machine, used in log list)
- `use-log-filters.ts` - Shared filter state schema (used in LogList, LogFilters, useInfiniteScroll - 3+ components)
- NOT extracted: Form validation logic (react-hook-form handles, not reused)
- NOT extracted: Simple useState toggles (modal open/close - too trivial)

**Pattern:**
```typescript
// Extract when: reused in 2+ components
export function useLogFilters() {
  return useQueryStates(logFiltersSchema)
}

// Keep inline when: single-use, simple
const [isOpen, setIsOpen] = useState(false)
```

**Benefits:** DRY principle for genuinely reusable logic, testable in isolation, clear separation of concerns, but avoids premature abstraction

---

**Topic 3: URL State Management Philosophy (Single Source of Truth)**

**Context:** Filter state, modal state, date ranges needed in URL for shareable links. Decision needed: how to manage URL state?

**Options Considered:**
- Option A: React state synced to URL with useEffect (two-way sync)
- Option B: URL as source of truth with nuqs library (read from URL, write to URL)
- Option C: Context/Zustand state with URL as secondary persistence

**Decision:** URL as single source of truth using nuqs library. All components read/write directly to URL state, no separate React state for URL-persisted values.

**Pattern (detailed in ADR-001):**
```typescript
// All components use same hook
const [filters, setFilters] = useLogFilters()

// Reads current URL params
console.log(filters.severity) // ['ERROR', 'CRITICAL']

// Writes to URL (triggers re-render in all consuming components)
setFilters({ severity: ['INFO'] })
```

**Benefits:** Shareable links work automatically, no sync bugs (single source of truth), components react to URL changes, history back/forward works correctly

**Detailed rationale:** See ADR-001 (Filter Reactivity Refactor) for full exploration of alternatives and consequences.

---

**Topic 4: Component Composition Patterns**

**Context:** Components need to be composable and maintainable. Decision needed: how to structure parent-child relationships?

**Options Considered:**
- Option A: Container/Presenter pattern (smart/dumb components)
- Option B: Feature-based components (LogCard contains all log rendering logic)
- Option C: Composition via children/slots (flexible but complex)

**Decision:** Hybrid - use presentation components for pure rendering, feature components when state/logic tightly coupled, composition via children for layout flexibility.

**Examples:**
- Presentation: `SeverityBadge` (pure props → JSX, no state)
- Feature: `LogDetailModal` (state + logic + presentation tightly coupled)
- Composition: `Card` component accepts children for flexible content

**When to split:**
- Split when: Component exceeds 200 lines, OR has multiple distinct responsibilities, OR presentation reusable elsewhere
- Keep together when: State and rendering tightly coupled (modal with form), OR component < 100 lines and cohesive

**Benefits:** Reusable presentation components, cohesive feature components, flexible composition where needed, pragmatic splitting (not dogmatic)

5. **Consequences:**

**Positive:**
- Consistent patterns across entire frontend codebase
- New features follow established patterns (reduced decision fatigue)
- SSR performance benefits (fast initial page load, SEO)
- URL state enables shareable links (requirement FILTER-07)
- Custom hooks testable independently from components
- Clear mental model for when to use each pattern

**Negative:**
- Learning curve for Next.js 15 Server/Client Component model
- nuqs library introduces dependency (but small, well-maintained)
- Some duplication accepted to avoid premature abstraction
- Pattern documentation needed (this ADR) to onboard new developers

**Neutral:**
- Patterns evolve as framework evolves (Next.js 15 → 16)
- Trade-offs between patterns sometimes unclear (document edge cases as encountered)

6. **Alternatives Rejected:**

**Full Client-Side (SPA):** Rejected because it loses SSR performance benefits. Initial page load requires JavaScript download + execution + API fetch before rendering anything. Next.js 15 Server Components enable instant HTML rendering. Requirement UI-05 mandates Server Components for data fetching.

**Full Server Components with Server Actions:** Rejected because it forces server roundtrips for UI state changes. Example: Modal open/close, filter checkbox toggle - these are pure client-side state changes, no server needed. Server Actions add 50-100ms latency for interactions that should be instant. Client Components better for interactivity per UI-06.

**Context/Zustand for Filter State:** Rejected because URL must be source of truth for shareable links (FILTER-07). Syncing URL ↔ Context is complex and bug-prone. ADR-001 documents attempted prop-based approach and why it failed. nuqs solves this cleanly by making URL the state store.

**Extract All Logic to Hooks:** Rejected because it creates abstraction overhead for simple cases. Example: `const [isOpen, setIsOpen] = useState(false)` for modal - extracting to `useModalState()` adds file, import, indirection with zero benefit. Extract only when reused or complex.

- nuqs library: https://nuqs.47ng.com/
