**Required sections:**

1. **Header:**
```markdown
# ADR-002: Use Cursor-Based Pagination for Log List

**Status:** Accepted
**Date:** 2026-03-20
**Deciders:** Development Team
```

2. **Context:**
- Problem: Log list needs pagination for 100k+ logs, traditional offset pagination degrades performance
- Technical context: FastAPI backend, PostgreSQL 18, SQLAlchemy async queries
- Performance requirement: Page 100+ must load in under 500ms (from TEST-03)
- Specific issue: OFFSET forces database to scan and discard rows (page 100 = skip 5000 rows)
- Constraint: Must work with live data (concurrent inserts shouldn't cause duplicate/missing rows)

3. **Requirements Addressed:**
```markdown
- **LOG-02:** User can view paginated list of all logs
- **LOG-04:** Log list uses cursor-based pagination for constant query time
- **NFR-Performance:** Query execution time <500ms for pagination at any depth
- **TEST-03:** Tests verify database query performance with 100k+ logs
```

4. **Options Considered:**

**Option 1: Offset-Based Pagination (LIMIT/OFFSET)**
- Pros: Simple to implement, well-understood pattern, supported by all databases, allows jumping to arbitrary page
- Cons: Performance degrades linearly with page depth (page 1: 10ms, page 100: 500ms, page 1000: 5000ms), duplicate/missing rows if data changes, scans entire dataset to reach page N

**Option 2: Cursor-Based Pagination (keyset pagination)**
- Pros: Constant query time regardless of page depth (~10ms always), no duplicate/missing rows with concurrent inserts, works with live data, composite index enables efficient seeks
- Cons: Cannot jump to arbitrary page number, cursor becomes invalid if sort order changes, more complex implementation than offset

**Option 3: Search-After Pattern (Elasticsearch-style)**
- Pros: Similar performance to cursor-based, works well with full-text search
- Cons: Requires Elasticsearch infrastructure, overkill for relational database, adds operational complexity

**Option 4: Infinite Scroll with Window Functions**
- Pros: Good UX for time-series data, can use ROW_NUMBER() for pagination
- Cons: Window functions expensive on large datasets, doesn't solve OFFSET performance issue, complex sorting with multiple columns

5. **Decision:**
```markdown
We will implement cursor-based pagination using base64-encoded timestamp+id cursors.

Implementation pattern:
- Encode cursor from last item: `{"timestamp": "2024-03-20T15:30:00+00:00", "id": 123}` → base64
- Decode and query using composite comparison: `WHERE (timestamp, id) < (cursor_timestamp, cursor_id)`
- Composite index (timestamp DESC, id DESC) enables efficient seeks
- Base64 encoding keeps cursor format opaque to clients, allowing internal changes
```

Include code snippet from backend/app/utils/cursor.py (encode_cursor function, ~10 lines).

6. **Consequences:**

**Positive:**
- Query time constant (~10ms) at any page depth (page 1 = page 1000)
- Composite index (timestamp DESC, id DESC) enables efficient seeks
- No duplicate or missing rows even with concurrent inserts
- Opaque cursor format allows changing implementation without breaking clients
- Frontend useInfiniteScroll hook manages cursor state seamlessly

**Negative:**
- Cannot jump to arbitrary page number (no "go to page 50" functionality)
- Cursor becomes invalid if sort order changes (filter changes require new cursor)
- More complex implementation than offset pagination (~70 lines vs ~10 lines)
- Frontend must track cursor instead of page number

**Neutral:**
- Cursor must be included in URL state for shareable links (handled by nuqs)
- Requires composite index on every sortable field + id for stable ordering

7. **Alternatives Rejected:**

**Offset-Based Pagination:** Rejected because performance degrades to 5+ seconds at page 1000 with 100k logs. Testing requirement (TEST-03) specifies 100k+ log validation - offset pagination fails this requirement. Performance tests (test_logs_performance.py) verify page 100+ completes in <500ms, impossible with offset approach.

**Search-After Pattern:** Rejected because it requires Elasticsearch infrastructure. Project uses PostgreSQL for relational data - adding search engine adds operational complexity (deployment, monitoring, sync) without sufficient benefit. Full-text search needs met with PostgreSQL ILIKE for MVP.

**Window Functions:** Rejected because ROW_NUMBER() OVER (ORDER BY ...) forces full table scan even with indexes. Performance testing shows 2+ second queries at 100k scale. Doesn't solve the OFFSET performance problem, just wraps it differently.

**Required sections:**

1. **Header:**
```markdown
# ADR-003: Use BRIN + Composite B-tree Indexes for Time-Series Queries

**Status:** Accepted
**Date:** 2026-03-20
**Deciders:** Development Team
```

2. **Context:**
- Problem: Need efficient queries on 100k+ logs with time-range filtering, severity filtering, source filtering, and multi-column sorting
- Technical context: PostgreSQL 18, time-series data (timestamp always present), filter combinations (timestamp + severity + source)
- Performance requirements: List queries <500ms at page 100, analytics queries <2s with 30-day range
- Constraint: Indexes have storage cost (each index adds disk space), need balance between query speed and storage
- Data characteristics: Logs are append-only, timestamp column has natural ordering, 70/20/8/2 severity distribution

3. **Requirements Addressed:**
```markdown
- **DB-03:** Database has BRIN index on timestamp column for time-series queries
- **DB-04:** Database has B-tree indexes on severity and source columns
- **DB-05:** Database has composite index on (timestamp, severity, source) for filtered queries
- **NFR-Performance:** Query execution time <500ms for pagination at any depth
- **TEST-03:** Tests verify database query performance with 100k+ logs
- **ANALYTICS-06:** Analytics queries require date range filter (BRIN index optimizes)
```

4. **Options Considered:**

**Option 1: Single B-tree Index on Timestamp Only**
- Pros: Simple, covers time-range queries efficiently, standard approach
- Cons: Cannot optimize multi-column filters (timestamp + severity), forces table scan when filtering by severity alone, doesn't leverage time-series optimization

**Option 2: BRIN Index on Timestamp + B-tree on All Filter Columns**
- Pros: BRIN ultra-compact for time-series data (pages_per_range=128 means 1 index entry per ~1MB data), B-tree indexes cover severity/source filters, query planner chooses best index per query
- Cons: Multiple indexes to maintain, query planner might choose wrong index without composite option

**Option 3: Composite B-tree Index (timestamp, severity, source)**
- Pros: Single index covers all common filter combinations, efficient for multi-column queries, supports ORDER BY timestamp DESC
- Cons: Large index size (~8-10% of table size), doesn't optimize timestamp-only queries as well as BRIN, index unused if query filters only severity/source

**Option 4: Covering Index with INCLUDE Columns**
- Pros: Index-only scans (no table lookup), fastest possible queries
- Cons: Massive storage overhead (includes message text), PostgreSQL INCLUDE limited to B-tree only, overkill for log viewing use case

5. **Decision:**
```markdown
We will implement a hybrid indexing strategy combining BRIN and composite B-tree:

1. **BRIN index on timestamp:**
   - CREATE INDEX idx_logs_timestamp_brin ON logs USING BRIN (timestamp) WITH (pages_per_range=128, autosummarize=on)
   - Optimizes time-range queries (date_from/date_to filters)
   - Ultra-compact: ~0.1% of table size (vs 5% for B-tree)
   - Autosummarize keeps index updated automatically

2. **Composite B-tree index (timestamp DESC, severity, source):**
   - CREATE INDEX idx_logs_composite ON logs (timestamp DESC, severity, source)
   - Optimizes multi-column filtering (timestamp + severity, timestamp + source, timestamp + severity + source)
   - Supports ORDER BY timestamp DESC (default sort)
   - Enables efficient cursor pagination with composite key

3. **No individual B-tree indexes on severity/source:**
   - Composite index covers these columns
   - Reduces index maintenance overhead
   - Query planner uses composite index prefix for severity-only queries
```

6. **Consequences:**

**Positive:**
- BRIN index reduces storage by 98% compared to B-tree on timestamp (100MB table → 100KB BRIN index)
- Composite index handles all common query patterns (85% of queries use timestamp + filter)
- Query performance meets requirements: pagination <500ms at page 100, analytics <2s with 100k logs
- Autosummarize on BRIN eliminates manual index maintenance
- Index choices validated by performance tests in test_logs_performance.py

**Negative:**
- Two indexes to maintain (BRIN + composite B-tree)
- Composite index unused for severity-only or source-only queries (rare edge case)
- Index size ~5-6% of table size (acceptable trade-off for performance)

**Neutral:**
- Query planner chooses between BRIN and composite based on query pattern
- VACUUM required for BRIN summarization (already in standard PostgreSQL maintenance)

7. **Alternatives Rejected:**

**Single B-tree on Timestamp:** Rejected because it cannot optimize common multi-column queries (timestamp + severity). Performance tests show 800ms+ query times for filtered pagination vs 150ms with composite index. Requirement TEST-03 mandates <500ms performance.

**Covering Index with INCLUDE:** Rejected because storage cost is prohibitive (15-20% of table size including message text). Log messages average 100 bytes - including in index multiplies storage by 3x. Benefit (index-only scans) doesn't justify cost for read-heavy log viewing use case.

**GIN/GiST Indexes:** Rejected because these are designed for full-text search or geometric data. Project uses simple ILIKE for message search (acceptable performance for MVP). Adding GIN index for message search deferred to v2 (FEAT requirements).

**Separate B-tree Indexes:** Rejected because composite index provides superset functionality. Query planner uses leftmost prefix of composite index for severity-only queries. Three separate indexes (timestamp, severity, source) would triple maintenance overhead with minimal benefit.
