# Phase 05.3: Migrate Create Log Form to Modal - Research

**Researched:** 2026-03-26
**Domain:** React form component refactoring, modal patterns, URL state management
**Confidence:** HIGH

## Summary

Phase 05.3 migrates the create log form from a standalone `/create` page to a modal accessible from the `/logs` page header. This aligns with the edit/delete modal pattern established in Phase 05.2, creating a consistent modal-based CRUD experience across all operations.

The existing `CreateForm` component is well-architected and can be adapted for modal usage through optional callback props (`onSuccess`, `onCancel`). The form currently uses react-hook-form with Zod validation, submission overlay, and router.push navigation—all patterns that can be conditionally controlled based on prop presence.

Technical approach: Add "+ Create Log" button to logs page header, use nuqs `useQueryState('create', parseAsBoolean)` for URL-controlled modal state (matching the `?log=123` pattern), wrap CreateForm in Dialog component, and remove the `/create` route entirely. The form adapts behavior based on callback presence: modal context closes modal and triggers refresh, page context navigates to /logs.

**Primary recommendation:** Reuse existing CreateForm with optional callback props for context-aware behavior. This maintains single source of truth for create logic while enabling both page and modal usage patterns.

## Architecture Patterns

### Pattern 1: Optional Callback Props for Context Adaptation
**What:** Form components accept optional callback props to adapt behavior for different usage contexts
**When to use:** When same form logic needed in both standalone page and modal contexts
**Example:**
```typescript
// Source: Established pattern from Phase 05.2 EditForm component
interface CreateFormProps {
  onSuccess?: () => void
  onCancel?: () => void
}

export function CreateForm({ onSuccess, onCancel }: CreateFormProps = {}) {
  const router = useRouter()

  const handleSuccess = () => {
    toast.success('Log created successfully')
    if (onSuccess) {
      onSuccess() // Modal: close and refresh
    } else {
      router.push('/logs') // Page: navigate
    }
  }

  // Show overlay only when NOT in modal context
  const showOverlay = !onSuccess && isSubmitting

  return (
    <div className="relative">
      {showOverlay && (
        <div className="absolute inset-0 bg-white/60 backdrop-blur-sm flex items-center justify-center rounded z-50">
          <div className="bg-white px-6 py-3 rounded-lg shadow-lg border flex items-center gap-3">
            <div className="animate-spin h-5 w-5 border-2 border-slate-300 border-t-slate-600 rounded-full" />
            <span className="text-sm font-medium text-slate-700">Creating log...</span>
          </div>
        </div>
      )}

      <form onSubmit={form.handleSubmit(onSubmit)} className="space-y-6 max-w-2xl">
        {/* Form fields... */}
      </form>
    </div>
  )
}

// Usage in modal context:
<CreateForm onSuccess={handleCreateSuccess} />

// Usage in page context:
<CreateForm />
```

### Pattern 2: URL-Controlled Modal State with nuqs
**What:** Modal open/close state synchronized with URL query parameter for deep linking and navigation
**When to use:** All modal dialogs that should support direct linking and browser back/forward
**Example:**
```typescript
// Source: Existing pattern from LogDetailModal (Phase 05.2)
import { useQueryState, parseAsBoolean } from 'nuqs'

export function LogsPage() {
  const [isCreating, setIsCreating] = useQueryState('create', parseAsBoolean)

  const handleCreateSuccess = () => {
    toast.success('Log created successfully')
    setIsCreating(false) // Close modal (clears ?create=true)
    onRefetch() // Refresh log list
  }

  return (
    <div>
      <Button onClick={() => setIsCreating(true)}>
        <Plus className="h-4 w-4 mr-2" />
        Create Log
      </Button>

      <Dialog open={isCreating} onOpenChange={(open) => !open && setIsCreating(false)}>
        <DialogContent className="max-w-2xl">
          <DialogHeader>
            <DialogTitle>Create Log</DialogTitle>
          </DialogHeader>
          <CreateForm onSuccess={handleCreateSuccess} />
        </DialogContent>
      </Dialog>
    </div>
  )
}
```

**Why this pattern:**
- URL reflects UI state: `?create=true` shows create modal
- Browser back/forward works naturally (close modal = back)
- Direct links supported: `/logs?create=true` opens modal directly
- Consistent with existing `?log=123` pattern for detail modal
- nuqs handles edge cases (null/undefined parsing, type safety)

### Pattern 3: Dialog Component Without Footer Buttons
**What:** Modal dialogs for forms that handle submission internally, no external cancel/confirm buttons
**When to use:** Forms with built-in submit button, when outside click/Escape is sufficient for close
**Example:**
```typescript
// Source: shadcn/ui Dialog best practices + Phase 05.2 modal pattern
<Dialog open={isCreating} onOpenChange={(open) => !open && setIsCreating(false)}>
  <DialogContent className="max-w-2xl">
    <DialogHeader>
      <DialogTitle>Create Log</DialogTitle>
    </DialogHeader>
    {/* No DialogFooter - form has its own submit button */}
    <CreateForm onSuccess={handleCreateSuccess} />
  </DialogContent>
</Dialog>
```

**Why:**
- Cleaner UI—one submit button in form (not duplicated in footer)
- Standard dialog UX—outside click closes without warning
- Form owns its own actions—better separation of concerns
- Matches edit modal pattern from Phase 05.2

### Pattern 4: Header Action Buttons Layout
**What:** Page header with title (left) and action buttons (right), primary action comes first
**When to use:** Any page with multiple actions in header
**Example:**

**Why:**
- Primary action positioned first (standard UI convention)
- Visual hierarchy: most important action most prominent
- Consistent with existing Export button placement
- Gap between buttons for visual separation

### Anti-Patterns to Avoid

- **DON'T use explicit `isModal` boolean prop:** Optional callbacks are cleaner—`onSuccess` presence implies modal context
- **DON'T duplicate form logic:** Reuse existing CreateForm, don't create separate ModalCreateForm
- **DON'T add unsaved changes warning:** CONTEXT.md explicitly decided close without warning
- **DON'T add cancel button:** Outside click/Escape sufficient per CONTEXT.md decisions
- **DON'T show submission overlay in modal context:** Button spinner sufficient, overlay redundant in modal

## Common Pitfalls

### Pitfall 1: Form State Leaking Between Page and Modal Contexts
**What goes wrong:** Modal form retains values from previous submission when reopened
**Why it happens:** react-hook-form state persists across re-renders if not reset
**How to avoid:** CreateForm already uses defaultValues—ensure form.reset() called on success or use key prop on form
**Warning signs:** Opening create modal shows previously entered values

**Prevention:**
```typescript
const handleCreateSuccess = () => {
  toast.success('Log created successfully')
  setIsCreating(false) // Close modal first
  onRefetch() // Then refresh
  // form.reset() called internally by CreateForm on success
}

// OR use key prop to force remount:
<CreateForm key={isCreating ? 'open' : 'closed'} onSuccess={handleCreateSuccess} />
```

### Pitfall 2: Breaking Existing `/create` Page During Migration
**What goes wrong:** Removing `/create` route before modal is working breaks user flows
**Why it happens:** Incomplete migration—sidebar link points to non-existent route
**How to avoid:** Implement modal fully, verify working, then remove route and sidebar link in single commit
**Warning signs:** 404 errors when clicking "Create Log" in sidebar

**Safe migration order:**
1. Add "+ Create Log" button to logs page header
2. Implement create modal with CreateForm
3. Test modal functionality thoroughly
4. Remove sidebar link from layout.tsx
5. Delete `/create` directory
6. Verify no broken links remain

### Pitfall 3: Modal Not Closing After Successful Creation
**What goes wrong:** Create modal stays open after log created, showing empty form
**Why it happens:** onSuccess callback not clearing URL state (`setIsCreating(false)`)
**How to avoid:** Always call setIsCreating(false) in handleCreateSuccess before triggering refresh
**Warning signs:** Toast shows "Log created successfully" but modal doesn't close

**Correct order:**
```typescript
const handleCreateSuccess = () => {
  toast.success('Log created successfully')
  setIsCreating(false) // FIRST: Close modal
  onRefetch() // SECOND: Refresh list (shows new log)
}
```

### Pitfall 4: Overlay Showing in Modal Context
**What goes wrong:** Both submission overlay AND modal backdrop visible, confusing UX
**Why it happens:** CreateForm's existing overlay logic not conditional on context
**How to avoid:** Add conditional: `const showOverlay = !onSuccess && isSubmitting`
**Warning signs:** Double blur effect during submission in modal

**Fix:**
```typescript
export function CreateForm({ onSuccess, onCancel }: CreateFormProps = {}) {
  const isSubmitting = form.formState.isSubmitting
  const showOverlay = !onSuccess && isSubmitting // Only show in page context

  return (
    <div className="relative">
      {showOverlay && (
        <div className="absolute inset-0 bg-white/60 backdrop-blur-sm...">
          {/* Overlay content */}
        </div>
      )}
      {/* Form */}
    </div>
  )
}
```

- Preserves page context (filters, scroll position)

- Reduces navigation complexity (single source of truth)
