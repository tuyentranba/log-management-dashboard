## Summary

Phase 4.1 adds message search functionality to backend `/api/logs` and `/api/export` endpoints by implementing case-insensitive ILIKE filtering on the `message` field. This phase closes the gap between the existing frontend SearchInput component (implemented in Phase 3 with 400ms debouncing) and the backend which currently ignores the `search` parameter.

The implementation follows the exact pattern established in Phase 2 for source filtering (line 156 in logs.py). No new libraries or patterns are required - this is a straightforward addition of one WHERE clause using the ILIKE operator with `%{search}%` wildcard matching. The phase includes comprehensive test coverage mirroring the thoroughness of Phase 2's 25 integration tests.

**Primary recommendation:** Copy the source filter pattern verbatim (lines 153-156 in logs.py) and apply it to the message field. Add search parameter to both list_logs and export_logs_csv function signatures. Include search in export frontend integration per WYSIWYG principle.

### Anti-Patterns to Avoid
- **String concatenation without parameterization:** Never use f-string directly in WHERE clause (SQLAlchemy ILIKE handles parameterization automatically)
- **Adding validation for search length:** Frontend already has debouncing, backend should accept any non-empty search term
- **Creating separate search endpoint:** Keep search as filter parameter in existing /api/logs endpoint for consistency
- **Skipping export integration:** Export must support search to maintain WYSIWYG principle from Phase 4

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Text search matching | Custom SQL string manipulation | SQLAlchemy ILIKE operator | Handles SQL injection prevention, parameter binding, and database-specific syntax automatically |
| Query parameter validation | Manual request.query_params parsing | FastAPI Query with Optional[str] | Automatic validation, OpenAPI documentation, type coercion |
| Empty string handling | Explicit `.strip()` checks | Python's falsy check (`if search:`) | Handles None, empty string, and whitespace-only strings automatically |
| Test fixtures | Manual database setup in each test | Existing conftest.py fixtures | test_db fixture provides clean database per test with automatic cleanup |

**Key insight:** SQLAlchemy's ILIKE operator is specifically designed for case-insensitive pattern matching and handles all edge cases (SQL injection, NULL handling, parameter binding) that custom solutions would need to address. The Phase 2 implementation already validated this pattern for 100k logs.

## Common Pitfalls

### Pitfall 1: ILIKE Performance on Large Datasets
**What goes wrong:** ILIKE `%pattern%` queries cannot use B-tree indexes, resulting in sequential scans
**Why it happens:** Leading wildcard prevents index usage (databases can only use indexes for prefix matches)
**How to avoid:** Accept the tradeoff for MVP scale (100k logs), document future migration path to PostgreSQL full-text search (tsvector with GIN index) for 1M+ scale
**Warning signs:** Query times exceeding 500ms on search operations with 100k+ logs
**Evidence:** Phase 2 RESEARCH.md Pitfall 3 documents this tradeoff: "ILIKE bypasses B-tree indexes. Acceptable for MVP with 100k logs. For 1M+ scale, consider PostgreSQL full-text search (tsvector with GIN index)."

### Pitfall 2: SQL Injection via String Concatenation
**What goes wrong:** Using f-strings directly in SQL WHERE clauses allows SQL injection attacks
**Why it happens:** Developer might write `query.where(f"message ILIKE '%{search}%'")` thinking it's cleaner
**How to avoid:** Always use SQLAlchemy's ILIKE operator method: `query.where(Log.message.ilike(f"%{search}%"))` - SQLAlchemy handles parameterization
**Warning signs:** Using raw SQL strings with f-string interpolation instead of SQLAlchemy operators
**Evidence:** SQLAlchemy 2.0 documentation explicitly recommends using operator methods for safe parameter binding

### Pitfall 3: Not Testing Combined Filters
**What goes wrong:** Search works alone but breaks when combined with severity/source/date filters
**Why it happens:** Query building order matters - incorrect WHERE clause composition can create unexpected AND/OR logic
**How to avoid:** Add integration test specifically for `search + severity + source + date_range` combination
**Warning signs:** Tests only cover search in isolation
**Evidence:** Phase 2 test suite includes test_list_logs_combined_filters (test_logs_list.py) - same pattern needed for search

### Pitfall 4: Forgetting Export Endpoint
**What goes wrong:** Search works in list view but export ignores search parameter, violating WYSIWYG principle
**Why it happens:** Export endpoint has separate filtering logic (lines 292-315) that must be kept in sync with list endpoint
**How to avoid:** Add search parameter to both list_logs AND export_logs_csv, verify with integration test in test_export.py
**Warning signs:** test_export.py has no search-related tests
**Evidence:** Phase 4 CONTEXT.md emphasizes "WYSIWYG principle: Export must respect all filters including search"

### Pitfall 5: Not Handling Special Characters in Messages
**What goes wrong:** Search breaks or returns incorrect results for messages containing quotes, newlines, or other special characters
**Why it happens:** Assuming ILIKE only works with plain text patterns
**How to avoid:** Test with messages containing commas, quotes, newlines, and verify csv.writer handles them correctly (already validated in test_export.py:320-339)
**Warning signs:** No test cases for special characters in search terms
**Evidence:** Existing test_export_csv_special_characters verifies handling of complex message content
