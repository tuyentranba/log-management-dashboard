## Technical Decisions

**1. ILIKE Pattern Matching**
- Chose SQLAlchemy ILIKE over full-text search for MVP simplicity
- Pattern: `Log.message.ilike(f"%{search}%")` provides case-insensitive partial matching
- Mirrors existing source filter implementation for consistency
- Note: ILIKE bypasses indexes but acceptable for MVP (documented in RESEARCH.md)

**2. Empty Search Handling**
- Empty strings and None values are ignored (no filter applied)
- Tested in test_list_logs_search_empty: returns all logs when search=""
- Matches user expectation: clearing search should show all logs

**3. Parameter Binding Security**
- Used SQLAlchemy's parameter binding automatically handles SQL injection
- Never use raw SQL string concatenation for search terms
- Pattern `f"%{search}%"` is safe - SQLAlchemy escapes parameters

## Known Limitations

**Performance:**
- ILIKE bypasses database indexes (full table scan on message column)
- Acceptable for MVP with 100k logs
- Future optimization: PostgreSQL full-text search (tsvector + GIN index)

**Search Scope:**
- Only searches message field (not severity, source, or timestamp)
- Frontend has separate filters for severity and source
- Combining search with those filters works correctly
