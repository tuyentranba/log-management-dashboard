# Phase 4.1: Add search logic in backend - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement message search functionality in backend `/api/logs` and `/api/export` endpoints. Frontend already sends `search` parameter (Phase 3 SearchInput component with 400ms debouncing), but backend currently ignores it. This phase closes the gap by adding case-insensitive ILIKE filtering on the `message` field, following the same pattern established for `source` filtering in Phase 2.

</domain>

<decisions>
## Implementation Decisions

### Search Parameter Handling
- **Parameter type**: Single string (matches source filter pattern)
- **Query parameter**: `search` (optional, string)
- **Case sensitivity**: Case-insensitive using ILIKE (consistent with source filter)
- **Wildcard pattern**: Wrap both sides `%{search}%` (matches anywhere in message)
- **Empty handling**: Ignore empty or whitespace-only strings (skip filter entirely, same as source)
- **Length restrictions**: No minimum character requirement (frontend debouncing already limits requests)

### Export Endpoint Integration
- **Support in /api/export**: Yes - add search parameter to export endpoint
- **Frontend integration**: Update `exportLogs()` function to include search in filter parameters
- **Code cleanup**: Remove "excluding search" comment from `frontend/src/lib/api.ts:69`
- **WYSIWYG principle**: Export must respect search filter just like other filters (severity, source, date range)

### Performance Considerations
- **ILIKE performance**: Accept Phase 2 decision - "ILIKE bypasses indexes (acceptable for MVP)"
- **No additional constraints**: No result limits, character minimums, or timeouts beyond existing patterns
- **Scale assumption**: 100k logs with ILIKE is acceptable (documented in Phase 2 RESEARCH.md Pitfall 3)
- **Future optimization note**: Document in CONTEXT.md for future reference - "Consider PostgreSQL full-text search (tsvector) for 1M+ log scale"

### Test Coverage
- **List endpoint tests**: Comprehensive integration tests covering:
  - Basic search match (single word)
  - Case-insensitive matching (search "ERROR" matches "error")
  - Partial matching (search "timeout" matches "connection timeout occurred")
  - No results (search term not found)
  - Special characters (quotes, newlines in messages)
  - Combined filters (search + severity + source + date range)
  - Empty search ignored (null/whitespace treated as no filter)
- **Export endpoint tests**: Dedicated tests in `test_export.py`:
  - Export with search parameter
  - Search combined with other filters (e.g., search + severity)
  - Verify search results appear correctly in CSV output
- **Test pattern**: Follow Phase 2 thoroughness (25 integration tests for list endpoint)

### Claude's Discretion
- Exact test data setup for special character scenarios
- Error message wording for validation failures
- Code organization within existing endpoint structure

</decisions>
