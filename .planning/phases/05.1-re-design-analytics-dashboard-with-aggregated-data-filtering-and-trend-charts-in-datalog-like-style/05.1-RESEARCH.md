## Summary

Phase 05.1 refines the Phase 5 analytics dashboard by adding proper spacing/padding following Tailwind's 8-point grid system and implementing a visual time range filter with segmented controls and inline date pickers. The user identified spacing issues ("no padding, looks unrefined") and requested dashboard-appropriate time filtering similar to the Logs page but more visual.

This is a UX polish phase with clear design specifications captured in CONTEXT.md. All decisions are locked—research validates implementation patterns rather than exploring alternatives.

**Primary recommendation:** Use Tailwind CSS utility classes exclusively for spacing (p-6, gap-4, gap-8, space-y-8), implement segmented control with shadcn Button components styled with custom grouping CSS, and manage filter state with nuqs for URL persistence matching existing Logs page patterns.

### Pattern 1: Tailwind 8-Point Grid Spacing

**What:** Use Tailwind spacing scale where each unit = 0.25rem (4px). Recommended multiples of 4 for consistency.

**When to use:** All spacing decisions in this phase.

**Mapping:**
- `p-4` = 16px (4 × 4px)
- `p-6` = 24px (6 × 4px) ← Used for Card internal padding
- `gap-4` = 16px ← Used between stat cards
- `gap-8` = 32px (8 × 4px) ← Used between major sections
- `space-y-8` = 32px vertical margin between children

**Example:**
```tsx
// Page container with 24px padding and 32px section gaps
<div className="container mx-auto p-6 space-y-8">
  <header>...</header>      {/* Gap 8 (32px) */}
  <Card className="p-6">...</Card>  {/* Gap 8 (32px) */}
  <div className="grid gap-4">...</div>  {/* 16px between cards */}
</div>
```

**Why this works:** Tailwind's default scale is designed for this pattern. No custom configuration needed.

### Pattern 2: Segmented Control with Buttons

**What:** Connected button group where only one button is active at a time, styled to appear as single control.

**When to use:** Time range preset selector.

**Implementation approach:**
```tsx
// Segmented control pattern
<div className="flex items-center rounded-lg border border-input bg-background p-1">
  {presets.map((preset) => (
    <Button
      key={preset.value}
      variant={selected === preset.value ? 'default' : 'ghost'}
      size="sm"
      onClick={() => handleSelect(preset.value)}
      className={cn(
        "flex-1 rounded-md",
        selected === preset.value && "bg-primary text-primary-foreground"
      )}
    >
      {preset.label}
    </Button>
  ))}
</div>
```

**Styling notes:**
- Outer container: `border border-input bg-background p-1` creates bounding box
- Buttons: `flex-1` makes equal width, `rounded-md` for inner corners
- Selected state: `variant="default"` provides filled background
- Unselected state: `variant="ghost"` provides subtle hover

**Reference:** iOS-style segmented controls, common in admin dashboards.

### Pattern 3: Inline Conditional Expansion

**What:** Show/hide additional controls below trigger without modal overlay.

**When to use:** Custom date range picker expansion when "Custom" button clicked.

**Example:**
```tsx
<div className="space-y-4">

  <SegmentedControl selected={range} onChange={setRange} />

  {/* Conditional inline expansion */}
  {range === 'custom' && (
    <div className="grid grid-cols-2 gap-4 pt-2">
      <div>
        <Label>From</Label>
        <Input type="datetime-local" value={from} onChange={...} />
      </div>
      <div>
        <Label>To</Label>
        <Input type="datetime-local" value={to} onChange={...} />
      </div>
    </div>

```

**Why better than modal:** Keeps context visible, no focus trap issues, simpler state management.

### Pattern 4: URL State Management with nuqs

**What:** Persist filter state in URL query parameters for shareable links and browser history.

**When to use:** All filter controls (time range, severity, source).

**Existing pattern (from Logs page):**
```tsx
// Hook definition (create use-analytics-filters.ts)
import { useQueryStates, parseAsString, parseAsArrayOf } from 'nuqs'

const analyticsFiltersSchema = {
  date_from: parseAsString,
  date_to: parseAsString,
  time_range: parseAsString.withDefault('7d'),  // Preset selection
  severity: parseAsArrayOf(parseAsString),
  source: parseAsString,
}

export function useAnalyticsFilters() {
  return useQueryStates(analyticsFiltersSchema)
}

// Component usage
const [filters, setFilters] = useAnalyticsFilters()

// Update single filter
setFilters({ time_range: '24h' })

// Update multiple filters atomically
setFilters({ date_from: '2026-03-20', date_to: '2026-03-25', time_range: 'custom' })
```

**Benefits:**
- Browser back/forward works correctly
- Users can bookmark/share specific dashboard state
- No local state synchronization bugs
- Consistent with Logs page UX

### Pattern 5: datetime-local Input Validation

**What:** HTML5 native date-time picker with ISO 8601 format.

**Format:** `YYYY-MM-DDTHH:mm` (e.g., `2026-03-25T14:30`)

**Validation:**
```tsx
<Input
  type="datetime-local"
  value={dateFrom}
  onChange={(e) => setFilters({ date_from: e.target.value })}
  max={dateTo}  // Can't be after end date
  required
  className="..."
/>
```

**Browser support:** Baseline widely available since October 2021 (all modern browsers).

**Timezone handling:** Input is timezone-naive. Backend expects ISO 8601 with timezone (YYYY-MM-DDTHH:mm:ss+00:00). Convert client-side:
```tsx
// Convert datetime-local to ISO with timezone
const localDateTime = '2026-03-25T14:30'
const isoWithTZ = new Date(localDateTime).toISOString()  // '2026-03-25T14:30:00.000Z'
```

**Alternative avoided:** Third-party date pickers (react-day-picker, react-datepicker) add bundle size and maintenance burden. HTML5 native sufficient for this use case.

### Anti-Patterns to Avoid

- **Custom CSS for spacing:** Don't create custom CSS classes for padding/margins. Use Tailwind utilities exclusively.
- **JavaScript-based show/hide animations:** Don't use `react-spring` or `framer-motion` for filter expansion. Simple conditional rendering sufficient.
- **Separate filter state and URL state:** Don't maintain filter state in both useState and URL. Use nuqs as single source of truth.
- **Over-abstracting segmented control:** Don't create generic reusable segmented control component. Inline implementation simpler for single use case.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Date arithmetic | Custom date math (adding days, parsing) | date-fns (subDays, parseISO, format) | Edge cases (DST, leap years, month boundaries) are complex |
| URL state sync | Custom router.push with manual param building | nuqs useQueryStates | Type safety, atomic updates, null handling, history management |
| Form validation | Manual input validation | HTML5 validation attributes (required, min, max) | Browser-native, accessible, no JavaScript needed |
| Responsive grid | Custom breakpoint logic with useMediaQuery | Tailwind responsive prefixes (sm:, md:, lg:) | SSR-safe, no hydration issues, consistent with design system |
| Button group borders | Custom CSS border logic | Tailwind border utilities with conditional classes | Maintainable, no specificity issues, theme-aware |

**Key insight:** This phase uses only existing tools in the stack. No new complexity. Focus on correct Tailwind class combinations and nuqs patterns proven in Logs page.

## Common Pitfalls

### Pitfall 1: Inconsistent Spacing Values

**What goes wrong:** Using arbitrary spacing values (p-5, gap-7, space-y-9) breaks visual rhythm.

**Why it happens:** Developers eyeball spacing instead of following design system.

**How to avoid:**
- Use only p-4, p-6, gap-4, gap-8, space-y-4, space-y-8
- All values must be multiples of 4 (1, 2, 4, 6, 8, 12, 16, 20, 24)
- CONTEXT.md specifies exact values—follow them

**Warning signs:** UI feels "off" with uneven spacing, design review flags inconsistencies.

### Pitfall 2: Tailwind Purge Missing Classes

**What goes wrong:** Dynamic class names (e.g., `bg-${color}-50`) get purged from production build.

**Why it happens:** Tailwind's JIT compiler only includes classes it finds in source code strings.

**How to avoid:**
- Use complete class names in source: `bg-blue-50`, `bg-yellow-50`, etc.
- For dynamic values, use safelist in tailwind.config.ts (already configured for severity colors)
- Never construct class names with template literals for color values

**Warning signs:** Styling works in dev but breaks in production build.

### Pitfall 3: datetime-local Timezone Ambiguity

**What goes wrong:** User selects "2026-03-25 14:00" in their local timezone, backend interprets as UTC, off by hours.

**Why it happens:** datetime-local is timezone-naive, backend expects timezone-aware ISO 8601.

**How to avoid:**
- Convert to ISO with timezone before API call: `new Date(localValue).toISOString()`
- Backend analytics endpoint requires ISO format with timezone
- Don't send raw datetime-local value directly to API

**Warning signs:** Dashboard shows wrong data for date ranges, especially for users in non-UTC timezones.

### Pitfall 4: Segmented Control Overflow on Mobile

**What goes wrong:** 6 preset buttons with text labels overflow on mobile, causing horizontal scroll or text truncation.

**Why it happens:** Button labels ("Last 24 hours") too long for mobile width.

**How to avoid:**
- Use responsive text: `<span className="hidden sm:inline">Last </span>24h`
- Or switch to vertical stack on mobile: `flex flex-col sm:flex-row`
- Or use dropdown on mobile, segmented control on desktop

**Warning signs:** Horizontal scrollbar appears on mobile, buttons squished.

### Pitfall 5: nuqs State Not Updating

**What goes wrong:** Call `setFilters({ time_range: 'custom' })` but component doesn't re-render with new state.

**Why it happens:** nuqs updates are asynchronous, component reads state before URL updates.

**How to avoid:**
- nuqs hook returns current URL state—trust it as single source of truth
- Don't mix nuqs state with local useState for same values
- If need optimistic updates, use nuqs options: `{ shallow: false }`

**Warning signs:** Clicking preset buttons doesn't update UI immediately, requires page refresh.

### Example 2: Segmented Control Implementation

```tsx
// Source: CONTEXT.md Area 2 + shadcn Button patterns
// File: frontend/src/app/analytics/_components/time-range-filter.tsx

'use client'

import { Button } from '@/components/ui/button'
import { cn } from '@/lib/utils'

const TIME_PRESETS = [
  { value: '1h', label: '1 Hour' },
  { value: '6h', label: '6 Hours' },
  { value: '24h', label: '24 Hours' },
  { value: '7d', label: '7 Days' },
  { value: '30d', label: '30 Days' },
  { value: 'custom', label: 'Custom' },
]

export function TimeRangeFilter() {
  const [filters, setFilters] = useAnalyticsFilters()
  const selected = filters.time_range || '7d'

  const handlePresetClick = (value: string) => {
    if (value === 'custom') {
      setFilters({ time_range: 'custom' })
    } else {
      // Calculate date_from and date_to based on preset
      const now = new Date()
      const dateFrom = value === '1h' ? subHours(now, 1)
        : value === '6h' ? subHours(now, 6)
        : value === '24h' ? subDays(now, 1)
        : value === '7d' ? subDays(now, 7)
        : subDays(now, 30)

      setFilters({
        time_range: value,
        date_from: dateFrom.toISOString(),
        date_to: now.toISOString(),
      })
    }
  }

  return (
    <div className="space-y-4">
      {/* Segmented control */}
      <div className="inline-flex items-center rounded-lg border border-input bg-background p-1">
        {TIME_PRESETS.map((preset) => (
          <Button
            key={preset.value}
            variant={selected === preset.value ? 'default' : 'ghost'}
            size="sm"
            onClick={() => handlePresetClick(preset.value)}
            className={cn(
              "flex-1 rounded-md",
              selected === preset.value && "bg-primary text-primary-foreground"
            )}
          >
            {preset.label}
          </Button>
        ))}
      </div>

      {/* Custom date pickers - inline expansion */}
      {selected === 'custom' && (
        <div className="grid grid-cols-1 md:grid-cols-2 gap-4 pt-2">
          <div className="space-y-2">
            <Label>From</Label>
            <Input
              type="datetime-local"
              value={filters.date_from?.slice(0, 16) || ''}  // ISO to datetime-local format
              onChange={(e) => setFilters({ date_from: new Date(e.target.value).toISOString() })}
              max={filters.date_to?.slice(0, 16) || ''}
              required
            />
          </div>
          <div className="space-y-2">
            <Label>To</Label>
            <Input
              type="datetime-local"
              value={filters.date_to?.slice(0, 16) || ''}
              onChange={(e) => setFilters({ date_to: new Date(e.target.value).toISOString() })}
              min={filters.date_from?.slice(0, 16) || ''}
              max={new Date().toISOString().slice(0, 16)}  // Can't be in future
              required
            />
          </div>
        </div>
      )}
    </div>
  )
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Custom CSS breakpoints with JS | Tailwind responsive prefixes (sm:, md:) | Tailwind 3.0+ (2021) | SSR-safe, no hydration mismatches |
| Radix UI direct usage | shadcn/ui preset components | 2023 | Faster setup, consistent styling, easier customization |
| React Router query params | nuqs for Next.js App Router | Next.js 13+ (2022) | Type-safe, App Router native, better DX |
| date-fns v2 | date-fns v3/v4 | v3 in 2023, v4 in 2024 | ESM-first, better tree-shaking, Unicode date token deprecation |

**Deprecated/outdated:**
- **query-string library:** Replaced by nuqs for Next.js App Router (better integration, type safety)
- **moment.js:** Replaced by date-fns (smaller bundle, immutable API, modern)
- **Custom CSS Grid framework:** Replaced by Tailwind Grid utilities (fewer lines, more maintainable)

## Open Questions

1. **Should severity/source filters have "All" option or clear button?**
   - What we know: Logs page has checkboxes (multi-select) and text input (source)
   - What's unclear: CONTEXT.md mentions severity/source dropdowns but doesn't specify UX
   - Recommendation: Match Logs page pattern (checkboxes for severity, text input for source) for consistency

2. **Should filter changes trigger immediate refetch or require Apply button?**
   - What we know: Logs page refetches immediately on filter change
   - What's unclear: Analytics may have longer query times, consider debouncing
   - Recommendation: Immediate refetch for presets (< 1s query), debounce custom date changes (500ms)

3. **Should time range persist across page refreshes?**
   - What we know: nuqs persists in URL, browser history includes filter state
   - What's unclear: Should default be 7d or last selected range?
   - Recommendation: URL state persists if present, otherwise default to 7d (matches current implementation)
